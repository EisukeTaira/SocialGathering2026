# バレー大会 Webアプリ 基本設計書

## 1. 文書概要

### 1.1 目的

本書は、バレー大会向け Webアプリの初期実装に必要な画面構成、データ構造、責務分割、
今後の拡張方針を定義するための基本設計書である。

### 1.2 対象範囲

- Index ページ
- Tournament ページ
- ScoreInput ページ
- Admin ページ
- 共通レイアウト、ナビゲーション、状態管理
- 将来の API 連携を見据えたデータ設計

### 1.3 前提条件

- 初期段階は静的な HTML / CSS / JavaScript で構築する。
- 大会形式は総当たりとする。
- 使用コートは A / B / C / D / E / F コートの 6 面とする。
- 参加チーム数は全体で 40 チームとする。
- 各チームは特定コートに固定で割り当てる。
- ScoreInput ページで入力する主項目は、コート名、チーム名、セットスコアとする。
- 試合は 1 セットマッチで実施する。
- 試合開始予定時刻は固定タイムテーブルで管理する。
- Admin ページに認証は設けない。
- スコア確定後の修正は Admin ページからのみ可能とする。
- Index ページでは途中経過の得点は表示せず、対戦カードと状態表示を優先する。
- 将来的に Node.js / Express または Google Apps Script と連携できる構成を前提とする。

### 1.4 補足事項

- 40 チームを 6 コートへ固定配属するため、コートごとのチーム数は均等ではない可能性がある。
- コートごとの詳細な所属チーム一覧は運営確定後に Admin 画面から登録できる前提とする。
- 固定タイムテーブルの具体的な時刻一覧は運営表をもとに別途登録する。

## 2. システム概要

本アプリは、大会運営者、審判、参加者が試合進行と結果を共有するためのフロントエンドである。
トップ画面で大会全体の進行状況を俯瞰し、コート別ページで試合順や勝敗を確認し、
審判が結果を入力し、運営者が試合登録や入力受付の制御を行う。

## 3. 利用者と利用目的

| 利用者 | 主な目的 |
| --- | --- |
| 参加者・観客 | 試合進行、タイムスケジュール、結果確認 |
| 審判 | 担当試合の結果入力 |
| 大会運営者 | 試合登録、受付制御、結果修正 |

## 4. 画面一覧

| 画面ID | 画面名 | 主な役割 |
| --- | --- | --- |
| P01 | Index | 大会全体のタイムスケジュールと各コートの進行状況表示 |
| P02 | Tournament | コート別の試合順、対戦カード、スコア、勝敗表示 |
| P03 | ScoreInput | 審判による試合結果入力 |
| P04 | Admin | 試合登録、受付開始・停止、結果修正 |

## 5. 画面遷移設計

### 5.1 基本遷移

- Index から各 Tournament ページへ遷移できる。
- Index から ScoreInput ページへ遷移できる。
- Tournament ページから他コートの Tournament ページへ切り替えできる。
- ScoreInput 完了後は同ページ内で入力完了表示を行い、必要に応じて次試合入力に進める。
- Admin ページでは一覧表示と編集操作を同一画面で完結させる。

### 5.2 ナビゲーション方針

- 全ページ共通のヘッダーまたはタブナビゲーションを配置する。
- 現在表示中のページが分かるアクティブ状態を表示する。
- コート別ページ切り替えはタブまたはセグメントコントロール形式で実装する。

## 6. 情報設計

### 6.1 表示単位

- 大会全体
- コート単位
- 試合単位
- セット単位

### 6.2 主要情報

- タイムスケジュール
- 現在試合中のコート情報
- 次試合情報
- 試合順
- チーム名
- セットスコア
- 勝敗
- 試合状態
- 入力受付状態
- コート固定所属情報

### 6.3 試合状態

試合状態は以下の区分で管理する。

| 状態値 | 意味 |
| --- | --- |
| scheduled | 未開始 |
| in_progress | 進行中 |
| completed | 終了 |
| locked | 結果確定後の編集制限状態 |

### 6.4 コート運用方針

- 各チームは 1 つのコートに固定所属する。
- 試合は原則として所属コート内での総当たりとする。
- コートごとのチーム数差により試合数が異なる場合は、固定タイムテーブル上で空き枠を許容する。
- 試合登録順は Admin 画面から管理できるものとする。

## 7. 画面詳細設計

### 7.1 Index ページ

#### 7.1.1 目的

大会全体の進行状況を一目で確認できるダッシュボードとして機能する。

#### 7.1.2 主な表示要素

- 大会名
- 当日のタイムスケジュール
- 全コートの進行サマリー
- 各コートの現在試合
- 各コートの次試合
- お知らせまたは運営メッセージ領域

#### 7.1.3 UI要件

- モバイルではコートカードを縦並びで表示する。
- PC ではコートカードを複数列で表示する。
- 現在進行中の試合は色やバッジで強調する。
- 試合開始前、進行中、終了済みを視覚的に区別する。
- 途中経過の得点は表示せず、対戦カードと試合状態を中心に表示する。

#### 7.1.4 補足

- 固定タイムテーブルは上部に時系列で表示する。
- 各コートカードには現在試合と次試合を最低 1 件ずつ表示する。

### 7.2 Tournament ページ

#### 7.2.1 目的

コートごとの試合順、スコア、勝敗を一覧で確認できるようにする。

#### 7.2.2 主な表示要素

- コート名
- 試合一覧
- 試合順番号
- 対戦カード
- セットスコア
- 勝者表示
- 試合状態

#### 7.2.3 UI要件

- コート切り替えタブを配置する。
- 試合行ごとに状態が分かるバッジを表示する。
- 勝者は視認しやすいラベルまたはアイコンで示す。
- 未入力の試合は空欄ではなく、未確定であることが分かる表現にする。

### 7.3 ScoreInput ページ

#### 7.3.1 目的

審判が担当試合の結果を迷わず入力できるようにする。

#### 7.3.2 入力項目

- コート名
- 対象試合
- チームA 名
- チームB 名
- セットごとのスコア

#### 7.3.3 入力補助

- コート選択後に対象試合候補を絞り込む。
- 対象試合選択後にチーム名を自動表示する。
- 1 セットマッチのため、入力欄は 1 セット分のみを標準表示とする。

#### 7.3.4 バリデーション

- 必須項目未入力時は送信不可とする。
- 数値以外のスコア入力は不可とする。
- 同一試合に対してチーム名が未設定の場合は送信不可とする。
- 同点結果は許可しない。
- 送信中はボタンを無効化し、多重送信を防止する。

#### 7.3.5 完了時挙動

- 入力完了メッセージを画面上に表示する。
- 一覧反映後、次の試合入力へ進める導線を提供する。

### 7.4 Admin ページ

#### 7.4.1 目的

大会運営者が試合登録と入力受付の制御を行うための管理画面とする。

#### 7.4.2 主な機能

- 試合登録
- 試合編集
- スコア修正
- 入力受付開始
- 入力受付停止
- コート所属チーム登録
- 固定タイムテーブルに基づく試合時刻設定

#### 7.4.3 管理対象項目

- コート名
- 試合順
- チームA 名
- チームB 名
- 試合状態
- 入力受付状態

#### 7.4.4 UI要件

- 試合一覧と編集フォームの責務を分けて表示する。
- 現在の受付状態を明示する。
- 操作結果はトーストまたはインラインメッセージで通知する。
- 誤操作防止のため、破壊的変更には確認操作を入れる。
- スコア修正は Admin 専用操作として明示する。

## 8. データ設計

### 8.1 データモデル一覧

| モデル名 | 用途 |
| --- | --- |
| court | コート情報管理 |
| team | チーム情報管理 |
| match | 試合情報管理 |
| score | セットスコア管理 |
| appState | 画面共通状態管理 |

### 8.2 match モデル案

| 項目名 | 型 | 説明 |
| --- | --- | --- |
| id | string | 試合ID |
| courtId | string | コート識別子 |
| order | number | 試合順 |
| teamA | string | チームA 名 |
| teamB | string | チームB 名 |
| status | string | 試合状態 |
| winner | string | 勝者チーム名 |
| scheduledAt | string | 開始予定時刻 |
| updatedAt | string | 最終更新時刻 |
| inputEnabled | boolean | ScoreInput 受付可否 |
| scoreLocked | boolean | スコア修正が Admin のみに制限されているか |

### 8.3 score モデル案

| 項目名 | 型 | 説明 |
| --- | --- | --- |
| matchId | string | 対象試合ID |
| setNumber | number | セット番号 |
| teamAScore | number | チームA 得点 |
| teamBScore | number | チームB 得点 |

### 8.4 court モデル案

| 項目名 | 型 | 説明 |
| --- | --- | --- |
| id | string | コート識別子 |
| name | string | コート名 |
| assignedTeams | string[] | 所属チームID一覧 |
| scheduleSlots | string[] | 固定タイムテーブル枠 |

### 8.5 appState モデル案

| 項目名 | 型 | 説明 |
| --- | --- | --- |
| selectedPage | string | 現在ページ |
| selectedCourtId | string | 選択中コート |
| noticeMessage | string | 画面通知メッセージ |
| lastUpdatedAt | string | 最終更新時刻 |
| scoreInputEnabled | boolean | 全体の入力受付状態 |

## 9. フロントエンド構成方針

### 9.1 ディレクトリ構成案

```text
src/
  app/
    main.js
    router.js
    store.js
  components/
    layout/
    navigation/
    cards/
    forms/
    tables/
  pages/
    index-page.js
    tournament-page.js
    score-input-page.js
    admin-page.js
  styles/
    base.css
    layout.css
    components.css
    pages.css
```

### 9.2 責務分割

- app は初期化、状態管理、ページ切り替えを担当する。
- components は再利用可能な表示部品を担当する。
- pages は画面単位の組み立てとイベント接続を担当する。
- styles は全体テーマとコンポーネント単位の見た目を担当する。

### 9.3 状態管理方針

- 初期段階では単純な JavaScript オブジェクトで状態を保持する。
- データ取得処理と描画処理は分離する。
- 将来的な API 差し替えに備え、データアクセス関数を画面実装から分離する。
- スコア修正権限は状態として保持し、ScoreInput 画面からは変更できないようにする。

## 10. 非機能要件

### 10.1 保守性

- 画面ごとの責務を明確に分ける。
- 共通 UI はコンポーネント化する。
- データ構造は画面依存にしない。

### 10.2 アクセシビリティ

- セマンティックな HTML を使用する。
- フォーム要素には label を関連付ける。
- キーボード操作で主要機能を使えるようにする。

### 10.3 レスポンシブ対応

- モバイルファーストで設計する。
- スマートフォンでも主要情報が 1 画面内で把握できる構成とする。
- PC では一覧性を高めるために多列レイアウトを使用する。

## 11. 将来拡張方針

- Google Apps Script 連携時は match と score を API 経由で取得・更新する。
- Node.js / Express を導入する場合でも、フロントエンドの描画 API は極力維持する。
- Admin 認証が必要になった場合は、管理機能を別ルートに分離する。
- 試合形式が総当たり以外に拡張された場合は、match モデルにラウンド情報を追加する。

## 12. 次工程で実施する内容

次工程では以下を実施する。

1. 画面モックを兼ねた静的フロントエンドの作成
2. 仮データによる 4 画面の表示実装

4. ScoreInput と Admin の入力処理設計の具体化
5. API 連携前提のデータアクセス層の切り出し

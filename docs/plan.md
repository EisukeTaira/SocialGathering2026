# バレーボール試合状況管理システム 詳細設計書

## 1. プロジェクト概要
本システムは、6つのバレーボールコートの試合進行状況をリアルタイムで管理・可視化するためのWebアプリケーションです。特に、エントリーチームの一覧から始まり、最終的な優勝者が決まるまでの「トーナメントの流れ」を直感的に把握できることを目指します。

## 2. 主要機能
### 2.1 リアルタイム試合状況（Overview）
- 6コート分の現在の対戦カード、スコア、審判チームを一覧表示。
- Googleスプレッドシートの更新を数秒間隔で自動反映。

### 2.2 垂直上昇型トーナメント表（Bracket）
- 下から上へ勝ち上がっていく「ピラミッド型」の樹形図スタイル。
- **勝者自動反映**: 入力システムでスコアを入力し、試合終了ボタンが押されると、自動的に次ラウンドの対戦カードへ名前が移動。
- **6チーム構成**: 1コートあたり6チームの場合、シード2チーム（初戦免除）を含む計5試合のトーナメント構成とする。
- **敗者の試合**: 敗者の試合は逆ピラミッド型で表示する。

### 2.3 エントリーチーム一覧（Team Entries）
- 各コートに参加する全チームをリスト表示。

### 2.4 管理システム（Admin）
- 各コートの対戦チーム名、スコア、審判、試合ステータスの更新。
- **入力許可コントロール**: 各コートの「スコア入力システム」に対して、操作の許可（OK）/禁止（NG）をリアルタイムで切り替える機能。

### 2.5 入力システム（Score Input）
- **認可チェック**: GAS API経由でAdminからの許可ステータスを確認し、許可されている場合のみスコア送信を可能にする。
- 各コートの得点状況（数値入力）を送信。
- 試合終了ボタンが押されると、GAS側で勝敗確定と次ラウンドへの反映処理を実行させる。
- モバイル端末からの操作も考慮したUI。

## 3. デザイン指針 (UI/UX)
- **カラーコンセプト**: 完全な「モノクローム（白・黒・グレー）」を採用。
- **美学**: ミニマルかつプレミアムな質感。太いボーダー、大胆なタイポグラフィ（Inter/Roboto）、微細なアニメーション。
- **視認性**: 遠くからでも、あるいはスマートフォンでも、現在の得点が瞬時に読み取れるコントラスト。
- **装飾**: 試合エリアには薄い背景色（例：淡いイエローベージュ `#fdfdf0`）をアクセントとして使用。

## 4. 技術・設計方針
- **SOLID原則の遵守**:
    - **Single Responsibility (単一責任)**: 各クラスやモジュール（APIの各機能やUIコンポーネント）は一つの役割のみを持つ。
    - **Open/Closed (開放閉鎖)**: 既存のコードを修正せず、拡張によって機能を追加できる設計（例：SVG描画エンジンのプラグイン化）。
    - **Liskov Substitution (リスコフの置換)**: 基底クラスを派生クラスで安全に置き換え可能にする。
    - **Interface Segregation (インターフェース分離)**: 不要なメソッドに依存させない（GAS APIの軽量化）。
    - **Dependency Inversion (依存性逆転)**: 上位モジュールが下位モジュールに依存せず、抽象に依存する（API通信層の抽象化）。
- **クリーンコード**: 命名の明確化、適切なコメント、DRY（Don't Repeat Yourself）の徹底。

## 5. 技術アーキテクチャ
- **データ基盤**: Googleスプレッドシート（DBとして利用）。
- **通信層**: Google Apps Script (GAS) Web App API (`doGet` / `doPost`)。
- **ロジック層**: サーバーサイド（GAS）でトーナメント勝ち上がり判定、スコア連動を実行。
- **フロントエンド**: HTML5 / CSS3 / Vanilla JavaScript。
- **描画エンジン**: 
    - レイアウト: CSS Flexbox / Grid。
    - 接続線: SVG (Scalable Vector Graphics)。

## 6. データ構造（スプレッドシート）
- **Matchesシート**: 現在の試合状況（コート別）。チームA/Bそれぞれの得点カラム（数値）を持つ。
- **Bracketシート**: トーナメント構造。各試合のスコア（A/B個別）、勝者名、次の試合ID、次のスロット（A/B）を保持。
- **ScoreInputシート**: 各コートのリアルタイム入力用。得点A/B、審判、ステータス、およびAdminからの操作許可フラグ（OK/NG）。
- **Teamsシート**: コート別の参加チーム一覧。

## 7. 開発フェーズ
1. **設計・要件定義**: 現在の設計書（plan.md）およびGAS仕様書の合意。
2. **基盤実装 (Backend)**: GAS APIの構築（読み取り・書き込み・勝ち上がりロジック）と通信テスト。
3. **UI/UX実装 (Frontend)**: モノクロデザインの適用とレスポンシブ対応。
4. **機能実装**: 
    - Admin/ScoreInput 画面の構築。
    - トーナメント描画ロジック（SVG）とAPIの統合。
5. **テスト・調整**: 実機での動作確認と表示の微調整。
